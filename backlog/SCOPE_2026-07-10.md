## 4. Milestone 1 — Minimum Viable Release Candidate

Phases beyond Track 0 needed to reach a shippable, deployable, correct minimum that a non-programmer could stand up. **Bugs first.**

**M1-A — Cage-card & print-control correctness (bug-led).** Files: `cagecard_gen5rs.php`, `cagecard_gen5rs-blindgeno.php`, `cagecard_printer.php`.
- **#29 (Bug, lead):** cage-card generators omit cages containing mice without alleles — direct fallout of the shipped #24 change. MVP ships broken/incomplete cards without this. Ensure allele-less animals/cages render.
- **#30 (control placement + reorg):** on `cagecard_printer.php` the card-color filter is wired to the wrong stage — it filters the cards passed to the generator rather than the cages selectable for printing. Correct the control (move it near the generate buttons / make it do what its label implies) and complete the related page-control layout reorg in the same pass.

**M1-B — Data-entry guards (#25).** Files: `add_animals.php`, line/allele management.
- Enforce/confirm the DOB requirement in `add_animals.php`.
- Characterize and guard allele↔line side-effects when alleles are added/removed/re-added after animals exist on a line (records the failure mode M1-E repairs).

**M1-C — Dependency documentation (#18).** Deliverable: a dependencies doc covering each external resource (Apache, PHP; FPDF, Tabler icons; PHPMailer) with purpose, license, and self-containment status (redistributed / user-provided / requires internet). Gates a license-compliant, air-gap-capable on-prem release — an MVP deliverable, not a nicety. Feeds M1-H.

**M1-D — Reactive UI (#26).** Remove manual REFRESH dependencies; update pages on interaction rather than requiring an explicit refresh (e.g. `cage_location.php` should refresh on the add-cage action). Scope the sweep to the pages that currently need a manual refresh after a mutation.

**M1-E — Admin repair surface (#28).** Admin-gated UI to backfill `table_genotypes` when a line gains an allele after animals were created: review the impacted filtered selection, present genotype select boxes, confirm, commit. Extends the Phase G admin surface and closes the loop opened by #24/#25.

**M1-F — Concurrency behavior *documentation* (#27, investigation half).** Enumerate current multi-user lost-update behavior across the mutation pages and sort each into "document for the user" vs. "fix in M2." No fixes in M1 — but the MVP ships with the risk known and documented rather than silent. Feeds M2-A.

**M1-G — DB-engine portability (#37).** Reframes "MySQL 8.0 EOL → MariaDB" as **portability, not a forced swap**: make the schema and code run on both MySQL (8.0/8.4) and MariaDB (10.11+/11.x), with MariaDB as the *recommended default for fresh installs* and existing MySQL deployments left untouched.
- Rationale: MySQL 8.0.46 is the final 8.0 release (EOL April 2026); native MySQL 8 arm64 is unavailable on Raspberry Pi OS; and RPM/DEB packaging forbids MariaDB + MySQL co-install on one host — so portability both fixes EOL and avoids ever forcing a second engine onto a shared host. Existing-MySQL hosts keep MySQL; the Pi and fresh hosts use native MariaDB.
- Portability audit / test matrix: `utf8mb4_0900_ai_ci` collations → neutral substitute (single schema that loads on both, or a maintained MariaDB variant); `mysqldump`-generated version-gated view syntax in `mousebook_install_schema.sql`; auth-plugin defaults (`caching_sha2_password` vs `mysql_native_password`/`ed25519` — PHP 8.3 mysqli supports both); any MySQL-8-only functions in the filter/query builder. `MyISAM` usage is compatible with both.
- Non-goal: uninstalling MySQL where other apps depend on it. Multi-engine-on-one-host, if ever needed, is a documented container/second-instance recipe, not a native install.

**M1-H — Installation & release readiness (#36).** The MVP exit gate: confirm a non-programmer can stand Mousebook up on the reference environments.
- Validate the two reference test environments: **Raspberry Pi 5 (8GB+) on Raspberry Pi OS** and a **RHEL/CloudLinux VM** (GoDaddy-like). *DB engine per M1-G: MariaDB native on the Pi; existing MySQL or MariaDB on the RHEL/CloudLinux host.*
- Ensure codebase (pages, bash scripts, SQL starters) is current and consistent with M1-G's engine decision.
- Refresh the README and step-by-step deploy/management docs (with markup + images): environment setup (OS deps, httpd/PHP/firewall), clone, fill config files, deploy backend, first-run smoke test with the initial admin account (userbook + colony DB + email), userbook user management, animalbook colony management.
- Provide a deploy shell script that stands up a supported environment with **no deviations from the documented instructions**.
- Depends on M1-C (#18 dependency docs) and M1-G (#37 engine decision).

## 6. Ordering rationale

- **Track 0 before everything:** it is the merge gate. The credential cleanup (T0.2) and read-guard sweep (T0.3) are latent-defect closers on already-shipped code and should not wait behind a new phase. `autoclipsheet.php` (T0.7) rides in Track 0 with its new `index.php` launcher.
- **#29 leads Milestone 1** for the same reason #22/#23 led the last cycle: an active output defect on freshly shipped code (#24).
- **#26, #28, #30 promoted to Milestone 1** (owner decision): required for a viable release rather than deferred enhancements.
- **#37 (M1-G) is placed before #36 (M1-H)** and framed as portability: it settles the DB-engine story (EOL fix + Pi native install + shared-host coexistence) that the install docs and deploy script depend on.
- **#36 (M1-H) is the MVP exit gate:** an MVP that a non-programmer can install *is* the minimum bar, so its validation/docs/deploy-script belong in Milestone 1; deeper install ergonomics stay in M2.
- **#27 is split** so the MVP ships with multi-user risk *documented* (cheap, high-value) while the *fixes* — which need design — land in M2.
- **Cage-card work is bundled** (M1-A + M2-C) so the print files are touched deliberately rather than repeatedly.
- **Remaining features (#17, #31) sit in M2** per the bug > feature rule.

## 7. Working conventions (carried)

- Re-baseline before coding; re-confirm HEAD if upstream advances (it did again this cycle: `e9d67c9` → `3cf276e`).
- Fix-confirm-proceed: findings + scoping questions before cutting code.
- Surgical `git apply`-compatible patches; `php -l` on all changed files; behavioral harness for key scenarios; explicit deploy ordering.
- **Never `sed -i` server-side** (resets inode group ownership → silent whitescreens); use `cat tmpfile > file` or `git apply` on a clone.
- MySQL 8 (8.0.46) + PHP 8.3.6 is the canonical validation harness; `MYSQLI_REPORT_OFF` at bootstrap. *Add a MariaDB harness under M1-G to validate portability.*
