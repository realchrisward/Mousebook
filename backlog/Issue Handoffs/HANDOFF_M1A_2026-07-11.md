## #29 — cage-card generators omit allele-less mice (BUG, fixed)

**Root cause (single point).** The generators (`cagecard_gen5rs.php`,
`cagecard_gen5rs-blindgeno.php`) do **not** query the DB — each receives
`$cages = unserialize(base64_decode($_POST['cages']))` (gen5rs L246) from a hidden
field the printer emits (`base64_encode(serialize($cages))`, printer L509/L525).
So whatever the printer's on-screen query drops never reaches the cards. That
query (`cagecard_printer.php` L306) inner-joined `table_genotypes`
(`... join \`table_genotypes\` on \`table_animals\`.\`animalautono\`=\`table_genotypes\`.\`animalautono\` ...`).
Animals with **zero** `table_genotypes` rows — exactly the allele-less
FOUNDER/new-line animals that the shipped **#24** fix now permits — are dropped by
that inner join. Net effect: a cage of only allele-less mice produced **no card**;
a mixed cage rendered but **silently lost** its allele-less animals.

**Fix.** One token: inner `join \`table_genotypes\`` → **`left join`**. The NULL
path already degrades cleanly: `GROUP_CONCAT` over no genotype rows → NULL
`genorxn`/`genotype` → no `conversion_geno` match → NULL `genoshort` → the geno
field renders **empty** (confirmed desired behavior — no placeholder). No generator
edits required; fixing the source array fixes both `gen5rs` and `blindgeno`.

**Validated (MySQL 8.0.46, real DDL, exact L306 query).** Seeded B6 (has
allelegroup, white cards) + EXT1 (allele-less founder line, pink cards); cages
`B6-1` (pure allele-bearing), `EXT1-1` (pure allele-less), `MIX-1` (both);
genotype rows only for the B6 animals.
- **Pre-fix (inner join):** rows for `B6-1`/A1 and `MIX-1`/A3 only. `EXT1-1` and
  its animal absent entirely; A4 dropped from `MIX-1`. ✗ (reproduces #29)
- **Post-fix (left join):** all 4 animals present; `EXT1-1` restored; A2/A4 carry
  `genorxn/genotype/genoshort = NULL` (empty geno); all 3 cages present. ✓

## #30 — misplaced color control + print-queue color management (fixed + expanded)

Decision (owner): **keep** the color filter as a print-run scope (it prunes
`$cages` before serialize, as before) but **relocate** it in the UI, and expand
the color workflow.

Changes, all in `cagecard_printer.php`:
1. **Relocated** the `colorfilt` control out of the top *source-selection* filter
   table (removed its `<th>`/`<td>`) into a new `#colorcontrols` block at the
   bottom of `cage_selection_form`, immediately above the generate forms — so it
   reads as scoping the print run, not the source list. It stays **inside**
   `cage_selection_form` so its `onchange=submitForm()` value still round-trips.
2. **Colors-in-queue summary** (with per-color counts), e.g.
   `In queue: pink (1), white (2)`. Snapshotted from the **full** `$cages`
   *before* the in-place color prune (L371–380), so it lists every queued color
   even while one color is selected for printing. `(unset)` bucket covers lines
   with no `card_color`.
3. **"Show cages of color"** button → read-only readout listing the queued cageids
   for the selected color (post-prune `$cages` keys; whole queue when `all`).
   Non-destructive; leaves the queue multi-select alone.
4. **"Remove cages of color"** button → deletes the selected color's cages from
   `CagesForPrinting` (for the post-print cleanup workflow). New
   `isset($_POST['remcage_color'])` handler following the existing add/remove
   pattern. **`all` is blocked** with the message:
   *"Unable to remove by color 'all'. You may clear the entire queue using the
   bulk-remove cages button"* — so it can never nuke the whole queue by accident.

**Note on `card_color` location (caught in the harness).** `card_color` lives on
**`table_lines`** (per line, via `lineassignment`), not `table_cages`. The
remove-by-color DELETE therefore routes
`CagesForPrinting → table_cages → table_lines` and filters `table_lines.card_color`.

**Validated.**
- remove-by-color DELETE (`white`) removed `B6-1` + `MIX-1` (line B6 = white),
  left `EXT1-1` (pink). ✓ Scoped and correct.
- PHP logic unit-tested: counts `pink (1), white (2)` ✓; show-white → `B6-1, MIX-1`
  ✓; show-all → all 3 ✓; `all`-block message exact ✓; specific-color allowed ✓.

## Behavior notes / non-goals

- **Refresh lag (matches existing behavior).** The remove-by-color handler runs
  after the queue **listbox** is built (same as the existing add/remove buttons),
  so the queue listbox reflects the removal on the next load. The colors-in-queue
  summary and print payload derive from `$cages` (built after the handler), so
  they update immediately. This is the same one-cycle lag #26 (M1-D) addresses;
  **not** fixed here to avoid scope creep into #26.
- No schema/migration and no new config keys/dependencies. PHP-only.
