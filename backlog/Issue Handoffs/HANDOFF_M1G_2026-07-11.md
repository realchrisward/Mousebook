## What the audit found

**Exactly one incompatibility exists, and it is fatal.** MySQL 8's
`utf8mb4_0900_ai_ci` collation does not exist in MariaDB. It appeared in **13 places**
in `mousebook_install_schema.sql` (the 2 InnoDB reservation tables + 11 view blocks).
Loading the schema verbatim on MariaDB dies at line 460:

```
ERROR 1273 (HY000): Unknown collation: 'utf8mb4_0900_ai_ci'
```

The import **aborts partway**, leaving **25 tables, 0 views, 0 procedures** — a colony DB
that looks present but is silently broken. That is the whole of the MariaDB blocker.

**Everything else on the M1-G audit list is a non-issue** (all verified, not assumed):
- mysqldump version-gated syntax (`/*!50001`, `/*!50003`, `/*!50503`) executes fine on MariaDB.
- **No MySQL-8-only SQL** anywhere: no window functions, CTEs, `JSON_TABLE`, `ANY_VALUE`, `REGEXP_LIKE`.
- MyISAM (37) + InnoDB (2), `LOCK TABLES … WRITE`, mixed `latin1`/`utf8mb3`/`utf8mb4` charsets, and the cage-card `CONVERT(… USING utf8mb3)` genotype join — all behave identically on both engines.
- No `DEFINER=` clauses in the dumps → imports don't depend on a DB user existing.
- Auth plugins: PHP 8.3 `mysqli` handles both `caching_sha2_password` (MySQL default) and `mysql_native_password` (MariaDB default). No config change.
- The 10 stored procedures carry `utf8mb3_general_ci` — portable as-is.

## The fix

`utf8mb4_0900_ai_ci` → **`utf8mb4_unicode_ci`** at all 13 sites. One schema file that loads
on both engines. **No PHP changes were required** — no code references a collation.

## Migration (existing MySQL installs — recommended)

A live MySQL colony keeps working either way, **but the collation is baked into the stored
metadata of the 2 tables and 11 views**, so `mysqldump` re-emits it — and *that backup cannot
be restored onto MariaDB*. Applying the migration makes existing backups portable.

`migration_m1g_collation_portability.sql`: 2 `ALTER TABLE … CONVERT TO` + `CREATE OR REPLACE`
of all 13 views (definitions lifted byte-identically from the schema, anchored on the
`/*!50001 VIEW` blocks, not the placeholder stubs). Idempotent, touches no data, colony DBs
only (userbook was never affected). Includes a verification query block.

## Flagged for M1-H (#36) — found while testing, NOT fixed here

1. **`setup.sh` is stale to the point of broken.** It imports `default_userbook.sql`,
   `default_animalbook.sql`, `mousebook_migration_v1.sql`, `mousebook_migration_v2.sql`
   and runs `patch_auth.php` — **none of those five files exist in the tree.** It also
   probes `validate_password.policy` (MySQL-only) and hard-checks for `mysql-client`.
   Per your call, all of it stays in M1-H. *(MariaDB ships `mysql`/`mysqld`/`mysqldump`
   as compat command names, so the client check will pass on MariaDB — but the missing
   SQL files mean the import steps silently skip with warnings today.)*
2. **The install schema hardcodes its database name** (`mousebook_install_schema.sql`
   L32–33: `CREATE DATABASE IF NOT EXISTS animalbook; USE animalbook;`). Piping it into a
   differently-named colony DB **silently writes to `animalbook` instead** — which defeats
   `setup.sh`'s custom-DB-name prompt *and* the multi-colony-per-host theme in M2.
   Documented in `DB_ENGINE_SUPPORT.md` §5; a real bug, awaiting your go-ahead.
3. **`DB_ENGINE_SUPPORT.md` §0** is written as a plain-English admin summary and is meant
   to be lifted into the M1-H install docs (engine choice, Pi/arm64 rationale, the
   co-install conflict).

## Also worth knowing

**MySQL and MariaDB cannot co-install natively.** Verified with apt: installing
`mysql-server` **removes** `mariadb-server`. This is exactly the RPM/DEB conflict the
scope anticipated, and it's why portability (not a forced swap) was the right call —
on a shared host already running MySQL, Mousebook must use what's there.
