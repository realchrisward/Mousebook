## Design decision worth understanding: the ratchet

The engine/charset check compares against a **checked-in baseline**, not against an ideal.

The baseline currently records `MyISAM` / `latin1` / `utf8mb3` — because *that is what the schema
actually ships*. If the check asserted "everything must be InnoDB + utf8mb4," CI would be red from
its first run, and a permanently-red check is one people learn to ignore.

Instead it fails only on **drift**: an engine or charset that changed without anyone meaning it to.
When Track B-1 converts to InnoDB, the baseline is regenerated **in that same PR**, so the change is
a reviewable diff rather than a silent mutation. The check tightens as the schema improves, and it
never has to be disabled.

## Known limitation — read this before the first run

**The MySQL 8.0 job is unproven.** There is no MySQL server available in the development container,
so everything above was validated on MariaDB 10.11 only. The MySQL job is written to be correct, but
the first run on GitHub may need an adjustment (most likely candidate: the service-container
healthcheck, which has to work for both `mariadb:10.11` and `mysql:8.0` images).

If the MySQL leg goes red on the first run, check whether it failed at *wait-for-database* (a
workflow bug — my problem) or at *schema load* (a real dual-engine incompatibility — the exact thing
this job exists to find). The output distinguishes them clearly.

## Findings surfaced by building this (both need issues filed)

### 1. Fresh installs are not utf8mb4 — and the M1-G migration is not in the repo

The baseline is an inventory of what a fresh install actually creates:

- **`userbook`: 100% `latin1`** — all 5 tables, including `userpass` and `userdetail`.
- **Colony: 11 `latin1` + 24 `utf8mb3` + 2 `utf8mb4`** (only the two `reservations_*` tables are
  utf8mb4).
- **There is no `migrations/` directory on `master`, `dev`, or `phase3_revisions`.** The M1-G
  collation migration exists only as a patch produced in a past session — it is not in the tree.

So: existing installs may have been migrated by an artifact that no longer exists, and every *new*
install regresses to latin1/utf8mb3. Together with the missing `set_charset()` (zero occurrences
repo-wide), this is the complete mojibake story — an accented strain name will not round-trip.

**This raises the priority of B-2 and adds scope to B-1:** the InnoDB migration pass should convert
charsets at the same time (one `ALTER TABLE` per table does both), and the migration must be
committed to `migrations/` this time.

### 2. `php/tutorial/` ships 8 unauthenticated executable endpoints

Confirmed while inventorying PHP files for the lint job: `php/tutorial/tuto1.php` … `tuto7.php` plus
`makefont.php` are live, reachable, auth-free PHP endpoints that emit PDFs. They are FPDF's demo
files. Already captured as Track A-3; noting here because the lint job now covers them (they parse
fine — which is not a defense of shipping them).
