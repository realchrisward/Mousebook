## M1-E — genotype backfill (#28)

Implements the repair for side-effect #1 in `AUDIT_M1B_allele_line_sideeffects.md`:
when an allele group is added to a line that already has animals, those existing
animals have **no** `table_genotypes` row for it. `genotype_backfill.php`:

- **Admin-gated** (`mb_guard_admin()`), wired into `MB_NAV_ITEMS` next to Manage
  Alleles. Its submit is `button_backfill` (button_-prefixed) so the guard actually
  covers it.
- Admin selects a **line**; the page lists every **live** animal (`dod IS NULL`) on
  that line missing a `table_genotypes` row for an allele group currently assigned
  via `key_allelebyline`, using a `NOT EXISTS` gap query.
- Each gap gets an allele `<select>` — options pulled from `list_allele` bucketed by
  `sexspecific` (M/F/all), mirroring add_animals. Choices: a real allele, `unk`
  (record as unknown), or `— skip —` (defer this gap).
- **Idempotent insert**: for each submitted gap the handler re-checks
  non-existence per `(animalautono, allelegroup)` before inserting (there is still
  no unique key on that pair), so re-submitting or concurrent use never duplicates a
  row. The handler runs **before** the gap re-query, so filled gaps disappear in the
  same POST (M1-D lesson applied).
- **Out of scope** (by the audit): editing existing genotype rows — that stays with
  manage_animals. This surface only fills missing rows.

**Validated (MySQL 8.0.46, real DDL incl. the M1-B unique index):** seeded B6 with
two live animals created before "Cre" was assigned (+ one dead animal, + one animal
already carrying a Flp row). Gap query returned exactly A1/Cre, A2/Cre, A2/Flp —
correctly excluding the already-present A1/Flp and the dead animal. Backfill filled
all three; re-query showed 0 gaps; a second backfill inserted 0 (all skipped); final
`table_genotypes` count 4 with no duplicates.

## P6 — mutation-path guard gap (comprehensive fix)

**Root cause.** `mb_guard_write()` / `mb_guard_admin()` only neutralised POST keys
**prefixed** `button_`. Several mutations don't follow that convention and so
slipped past the tier guards entirely:
- **Write-tier bypass:** `addcage_single` (cagerole, cage_location, printer),
  `remcage_*`/`addcage_batch`/`remcage_batch`/`remcage_color` (printer),
  `addcage1..4_*`/`clear_cages` (manage_cages), `confirm_changes` (manage_animals),
  `confirm_litter` (litterlogger) — a **read-only** user could trigger these.
- **Admin-tier bypass (higher severity):** manage_alleles names its mutations with a
  trailing `button` (`addgenebutton`, `addallelebutton`, …), which don't start with
  `button_`, so a **non-admin** could edit the allele/gene library.

**Fix (one place).** Added a central `mb_is_mutation_key($k, $admin_scope)` in
`session.php` and routed both guards through it. It recognises: `button_*` (except
login/disco), `addcage*`, `remcage*`, `clear_cages`, `confirm_*`, and — in admin
scope only — a trailing `button`. This neutralises every affected page at once with
no form-field renames, and as a bonus also closes `confirm_animals` (add_animals).

**Design notes.**
- `generate_animals` / `generate_litter` are intentionally **not** gated: they only
  write transient, per-user-purged reservation rows; the persistent commit steps
  (`confirm_*`) are gated, which is what prevents unauthorised data creation.
- `submit_cages` (PDF generation) and read/preview actions (`showcage_color`,
  `export_csv`, `get_tempanimals`, filters/selections) are correctly left alone.

**Validated:** `mb_is_mutation_key` unit-tested over 25 real keys — every mutation
(including the admin `*button` family, in admin scope only) recognised; every
non-mutation (filters, selections, `generate_animals`, `submit_cages`, `export_csv`,
login/disco) left untouched. The admin `*button` keys are correctly **not** stripped
in write scope.
