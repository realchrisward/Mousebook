## 2. Configurable auth-DB name (cPanel)

The name was reachable from five places. It now comes from **one**: `config.php` →
**`'userbook_db'`**, defaulting to `userbook`.

- **`includes/auth.php`** gains `mb_userbook_db($config)` (resolves + validates: rejects
  anything that isn't `^[A-Za-z0-9_]{1,64}$` and falls back to `userbook`, because this
  value reaches a database name and a database name cannot be bound as a parameter) and
  `mb_define_userbook_db($config)` (pins `MB_USERBOOK_DB`, idempotent).
- Both `new mysqli(..., 'userbook')` sites in auth.php (L45/L148) now resolve from config.
- **`includes/usertoken.php`** — its hardcoded `define('MB_USERBOOK_DB','userbook')` is
  gone; `mb_userbook_conn()` resolves from `$config`. It now `require_once`s auth.php
  explicitly, **because `forgot_password.php` pulls in usertoken.php without it** — that
  would otherwise have been a fatal on the one page an unauthenticated user can reach.
- **`includes/session.php`** — `mb_session_bootstrap()` pins the constant on every
  request, which is what `nav.php:109` needs to decide whether to show Manage Users /
  Manage Databases (it has no `$config` in scope).
- **`pages/databases.php`**, **`manage_users.php`**, **`manage_databases.php`** — the
  remaining literals.
- **`setup.sh`** prompts for the auth DB name (default `userbook`) and writes
  `userbook_db` into `config.php`.

The only two `'userbook'` literals left in the tree are the deliberate fallbacks in
`mb_userbook_db()` and `nav.php`.

**Proven, not asserted:** a full install with auth DB `myacct_userbook` and colony
`myacct_colony` — login renders both buttons, the auth-DB context surfaces Manage Users
*and* Manage Databases, `mb_userbook_conn` resolves its `dbaccess` row (no "cannot reach
the user database"), and **all 16 pages return 200 with zero fatals**. The default-name
path was re-run as a regression and is unchanged.

## 3. Widened columns

In `mousebook_userbook_install_schema.sql`:

| Column | Was | Now | Why |
|---|---|---|---|
| `dbaccess.db_accesspw` | `varchar(45)` | `varchar(255)` | 45 chars collides with generated/policy-mandated passwords; truncation on insert means the colony can't be reached with the password you actually set. |
| `dbaccess.db_formurl` | `varchar(45)` | `varchar(255)` | 45 doesn't fit a real URL — `https://mousebook.example.edu/mousebook/index.php` is 48 on its own. |
| `dbaccess.db_name` | `varchar(45)` | `varchar(64)` | 64 is the engine's own database-name maximum, so this can no longer be the narrower limit. Matters on prefixing hosts. |
| `userdbaccess.db_name` | `varchar(45)` | `varchar(64)` | Joined to `dbaccess.db_name`; had to move with it. |

**`migrations/migration_m1i_userbook_columns.sql`** does the same for existing installs.
Every change is a widening, so nothing can truncate; `MODIFY COLUMN` is idempotent.

Verified against a real old-width auth DB: `45/45/45` → `64/255/255`, existing rows
intact, clean second run, and a **200-character password and a full absolute URL both
store and read back byte-exact**.

`setup.sh`'s password cap moves 45 → 255. It still stores `db_formurl` as a relative path
(`../index.php`) — an absolute URL now fits, but a relative one keeps working if the site
later moves or changes hostname. Say the word if you'd rather it store the absolute URL.

## Deploy sequence

1. Apply **one** of the two patches above → commit.
2. **Existing installs:** apply the widening migration to the auth database:
   `mysql -u <admin> -p userbook < migrations/migration_m1i_userbook_columns.sql`
   Safe to re-run, no data touched.
3. **Deployment order is load-bearing:** the migration goes **before** the patched PHP.
   The PHP itself is backward-compatible with the narrow columns, but an install that
   moves to a prefixed DB name or a long generated password will silently truncate
   against a 45-char column.
4. Existing installs need **no** config change unless their auth DB is not named
   `userbook` — `'userbook_db'` defaults correctly if absent.
5. Existing colonies keep their own locations/roles; the seed rows only affect new ones.
