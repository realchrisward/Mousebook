## Root cause (shared)

Each page built its display listboxes first and ran the mutation handlers
*afterward*, so a change landed in the DB only after the lists that should reflect
it were already rendered. The **REFRESH** button was a bare submit that re-POSTed
the page to force a re-query — a workaround, not a feature.

## Fix (best-fit per page, same effect)

- **`cagerole.php`** — relocated the `addcage_single` role-assignment UPDATE to run
  *before* the display queries (right after the filters are parsed; inputs
  `$cage_selection`/`$roleB_selection` are already available). Removed the REFRESH
  button. Now "cages already with this role" reflects the assignment in the same POST.
- **`cage_location.php`** — its location add/retire/restore handlers were already
  reactive; only the `addcage_single` cage-move lagged. Relocated that UPDATE to run
  before the location/cage listboxes. Removed the REFRESH button. Carried the P6
  note about the non-`button_*` guard gap with the relocated block (see below).
- **`cagecard_printer.php`** — no REFRESH button, but the same lag on the queue and
  source listboxes (flagged for #26 in the M1-A handoff). The five queue handlers
  include `addcage_batch`, which consumes `$cage_batchlist` built by the *pre-handler*
  source query — so instead of reordering handlers, I **append a post-handler rebuild**
  of `$cage_listbox` (queue) and `$sourcecage_listbox` (source). This is additive
  (no handler movement), preserves the pre-handler `$cage_batchlist` batch-add targets,
  and reuses the existing `$sql_where_text` for the fresh source query. The two extra
  queries run after the mutations, so add/remove/clear/remove-by-color all reflect
  immediately.

Approach was a **server-side reorder only** — the full-page-POST model is unchanged;
no AJAX (which would be a large, inconsistent change for an MVP).

## Behavior notes / non-goals

- **Connection lifecycle unchanged in spirit.** Each relocated/added block opens and
  closes its own `mysqli` connection, matching the file's existing per-query pattern.
  The pre-existing connection-leak style elsewhere in these files was left as-is.
- **P6 security item re-flagged, NOT fixed here.** `addcage_single` posts as a
  non-`button_*` name, so Phase F's `mb_guard_write()` tier gate does not neutralise
  it for read-only users. This is the separate "non-`button_*` mutation-path audit"; I
  preserved existing behavior (and kept `cage_location`'s `$dbconnected` gate) rather
  than change the guard surface under M1-D. Worth scheduling that audit before M1-H.
- **Cosmetic "selected" highlight** in the printer's rebuilt source list mirrors the
  original comparison exactly (no functional change to which cages appear).
