## What the investigation found

The headline: the app is **not** unprotected — `add_animals` and `manage_cages` have a
real reservation system using `LOCK TABLES … WRITE`. But the locks cover the wrong
span, and the two genuinely dangerous behaviors are silent ones elsewhere.

1. **Autonumber TOCTOU (§1) — fails safe.** The `max()` reads that compute the next
   animal number happen *outside* the `LOCK TABLES` window that guards the reservation
   INSERT. Reproduced: two users both read `max=100` and derived the identical range.
   The collision does reach the real insert (confirm trusts hidden `minauto`/`maxauto`
   POST fields and inserts `animalautono` explicitly, never re-validating), **but**
   `animalautono` is the PK, so the duplicate fails with **1062**, `$ins_error` is set,
   and the batch halts writing nothing. Loser retries and succeeds. **No data loss.**
2. **`reservations_animals` cannot dedupe per user.** No unique key on `user` (PK is the
   AUTO_INCREMENT `maxautono` itself), so the `ON DUPLICATE KEY UPDATE` clause never
   fires on user. Reproduced: one user accumulated **two** live reservation rows — the
   intended one-reservation-per-user invariant does not hold.
3. **`idno` ear-tag duplicates (§2) — SILENT, high severity.** Same TOCTOU, but `idno`
   has **no unique constraint**. Reproduced: two animals on the same line inserted with
   the same `idno`, **silently accepted**. The autonumber is caught by the PK; the ear-tag
   number is not.
4. **Animal-edit lost update (§3) — SILENT, highest severity.** `manage_animals`
   `confirm_changes` writes whole-row UPDATEs from POST, and the schema has **no**
   version/`updated_at`/modified column anywhere (verified). Two people editing the same
   animal → last save silently overwrites the first. This is the core M2-A target.
5. **Shared print queue (§5).** `CagesForPrinting` has no `user` column — it's one global
   queue per colony DB. Surprising but not lossy.
6. **Purge is correctly per-user (§6).** Reproduced: one user loading Add Animals leaves
   another's live reservation intact. No issue.

| Reservation purge scope | — | **No issue** |

**Nothing here blocks the MVP.** The one path that could corrupt animal records fails
safe; the two silent risks are documented for users and carried into M2-A.

The doc includes a plain-English "Summary for users" table at the top (suitable for
lifting into the M1-H user documentation) and an **M2-A implementation checklist**:
optimistic locking on animal edits; atomic number claim (move `max()` inside the lock,
unique key on `reservations_animals.user`, consider `UNIQUE (line, idno)` + dedupe
migration); per-user print queue; human-readable conflict messages instead of raw 1062.
