**Baseline:** `master` @ `c588501` (Merge PR #43 from `dev`)
**Milestone framing:** Milestone 1 (MVP) is functionally complete. This document scopes the work
between MVP and **v2.0 — Mousebook Enhanced**, plus the small block of MVP close-out that is still
honestly outstanding.

**Supersedes:** nothing. Complements `BACKLOG_2026-07-12.md` (prioritized issue list) and

---

## 1. The organizing insight

Four separate items on the board — **#27a** (optimistic locking), **#27b** (atomic `idno` claim),
the **audit trail**, and **CSRF** — are all blocked on the same two structural facts:

1. **There is no transaction.** 35 of 37 colony tables are MyISAM.
2. **There is no single place where a write happens.** `new mysqli(...)` is constructed ad hoc in
   ~18 page files, and each page issues its own `UPDATE`s.

Fix those two things once and all four items become tractable. Fix them later and all four get built
twice — once around the limitation, once without it.

**Therefore: foundation first.** Nothing downstream of Track B should start before Track B lands.

```
Track A (hygiene)  ──┐
                     ├──► Track B (foundation) ──► Track C (correctness)
Track F (CI) ────────┘                         ──► Track D (audit)
                                               ──► Track E (hardening)
                                               ──► Track G (perf/UX)
```

## 4. Track B — Foundation (the gate)

### B-1 — MyISAM → InnoDB migration

**Verified feasible, empirically:** all 35 MyISAM tables convert with **zero failures** on MariaDB
10.11 (DYNAMIC row format). Zero FULLTEXT/SPATIAL indexes; every table already has a PRIMARY KEY.
The plausible blocker — `table_cages` declaring ~12 KB of varchars against InnoDB's ~8 KB in-row
limit — does **not** bite, because DYNAMIC pushes long varchars off-page.

**Measured cost** (100k animals / 50k cages / 300k genotypes, realistic string lengths):

| | Size |
|---|---|
| Colony data, MyISAM | 52 MB |
| Colony data, InnoDB | **100 MB** |
| `mysqldump` (either engine) | 40 MB / 5 MB gzipped |

~1.9×, on a colony larger than any real one. **Accepted as worth the cost.**

**Migration constraints (confirmed):** live installs exist, largest is **< 5k animals**; downtime
tolerance is **hours (weekend window)**; migration is run as a **shell script on the host**.

**Measured at that scale** (5k animals / 2.5k cages / 15k genotypes): the full 35-table conversion
takes **< 1 second**, and the colony occupies 10 MB. The downtime budget exceeds the requirement by
~1000×.

**Consequence: the migration tool is simple.** No online DDL, no chunking, no rollback-in-flight
machinery, no web UI. A synchronous shell script — consistent with the existing `setup.sh` pattern —
is the entire deliverable. Do not over-engineer this.

Deliverables:
- Install schema updated: fresh installs are InnoDB from the start.
- **`migrate_innodb.sh`** (shell, run on the host, matching `setup.sh` conventions):
  1. **Mandatory backup gate** — refuse to run without a verified fresh dump; take one if absent.
  2. Pre-flight: report engine mix, row counts, and **any `(line, idno)` duplicates** that would
     break C-3.
  3. `ALTER TABLE ... ENGINE=InnoDB` for each MyISAM table.
  4. Post-verify: 37/37 InnoDB, row counts identical pre/post; abort loudly on any mismatch.
  - Idempotent and safe to re-run (a re-run on a converted colony is a no-op that reports success).
- `BACKUP.md` rewritten: `--single-transaction` flips from *wrong* to *correct* — **but only if every
  table converts.** A partial migration keeps the caveat and gains nothing; this is why all 35 go in
  one pass.
- `DB_ENGINE_SUPPORT.md` updated; **re-verify on MySQL 8.0.46** (the conversion above was proven on
  MariaDB only — this must be confirmed before merge).
- `LOCK TABLES` call sites converted to transactions: `php/add_animals.php:71,178`;
  `php/manage_cages.php:80,412,842`. (Note the existing incoherence: `reservations_animals` is
  *already* InnoDB yet is guarded with `LOCK TABLES`.)
- Document the `AUTO_INCREMENT` semantic change (InnoDB can reuse IDs of deleted trailing rows after
  a restart; low exposure since animals are soft-deleted, but it is a real difference).

### B-2 — `mb_connect()`: one connection helper

Replace ~18 ad-hoc `new mysqli(...)` sites with a single helper that:
- connects,
- **calls `set_charset('utf8mb4')`** — currently **zero** occurrences repo-wide, which is a live
  mojibake bug for any non-ASCII line/strain/comment after the M1-G collation migration,
- sets the per-request actor context for the audit trail,
- centralizes the `debug_mode` block (currently copy-pasted into ~10 files, and it renders PHP
  warnings and SQL to the browser when left on in production).

### B-3 — `mb_write()`: one write chokepoint

```php
mb_write($conn, $table, $pk_col, $pk_val, array $new_values, $actor, $request_id);
```

Reads the before-image, diffs, writes the record **and** its audit rows in **one transaction**.
Unchanged columns are neither written nor logged.

This single helper is the precondition for #27a (one place to check the version token), the audit
trail (one place to log), and CSRF (one place to verify the token). **Build it once, deliberately.**

**Exit criteria for Track B:** every colony table is InnoDB on both engines; every page connects
through `mb_connect()`; a non-ASCII strain name round-trips correctly; `mb_write()` exists and
`manage_animals` uses it.

## 5. Track C — Correctness (#27, the only silent-data-loss work)

| Task | Detail |
|---|---|
| **C-1 (#27a)** | **Optimistic locking on animal edit.** Today: last save silently wins, and the earlier user's edits vanish with no warning (`CONCURRENCY.md` §3). Add a version / `updated_at` token; reject stale saves with a clear "this record changed while you were editing" path. **Top priority of the whole document.** |
| **C-2 (#27b)** | **Atomic `idno` claim.** Ear-tag numbers can silently duplicate. Derive `idno` inside the same atomic claim as the autonumber. With InnoDB this becomes a transaction rather than a `LOCK TABLES` dance. |
| **C-3** | **`UNIQUE KEY (line, idno)` on `table_animals`.** C-2 fixes the race *in code*; the constraint makes the invariant hold regardless of code path, forever, including the future import path. **Caveat:** existing colonies may already contain duplicates produced by the very race C-2 fixes — the migration must detect and report them before adding the constraint. |
| **C-4** | **Index gaps.** `table_animals` has no index on `dod` — and essentially every list in the app filters `dod is null`. Add `KEY (dod)`, `KEY (line, idno)`, consider `KEY (currentcage, dod)`. Drop the redundant `animalautono_UNIQUE` (the PK already enforces it; it is a second B-tree maintained for nothing). |
| **C-5 (#27c)** | **Print queue is global, not per-user.** Scope `CagesForPrinting` per user. Confusing rather than corrupting — lower priority, but cheap once B-3 lands. |

## 6. Track D — Audit trail

Full design in `ISSUE_audit_trail.md`. Summary: DB triggers are **unavailable** (the `TRIGGER`
privilege can't be relied on under CloudLinux shared hosting), so the audit write lives in
`mb_write()` (B-3), inside the same transaction as the record change (B-1).

Schema is measured, not guessed: interned actor/field lookups + `binary(16)` request id + two
indexes = **193 MB per 1M change rows**, vs 305 MB for the naive shape. `row_pk varchar(64)` is free
and works for both `animalautono` and `cageid`.

**The audit log will outgrow the colony it describes** (193 MB/1M rows vs 100 MB for a 100k-animal
colony). Ship the archive step and the growth-rate documentation *with* the feature, not after.

Sequence: tables + INSERT-only grant → `mb_write()` logging → `manage_animals` first (same pass as
C-1) → remaining tables → per-record History panel → admin Audit page → export → archive.

## 7. Track E — Hardening tail

| Task | Detail |
|---|---|
| **E-1** | **CSRF.** Zero references repo-wide. `SameSite=Lax` (`includes/session.php:100`) blocks the cross-site POST case in current browsers, so this is **defense-in-depth, not an open door** — but note that session auth (Phase F) is what created the exposure: when credentials were posted every request, they incidentally *were* a CSRF token. Verify a per-session token in `mb_write()`. |
| **E-2** | **Finish the SQL parameterization sweep.** 39 `prepare()` sites vs 66 `real_escape_string` and ~83 interpolated `SELECT`s (e.g. `php/manage_roles.php:80`). Mixed discipline is the condition under which the next injection is introduced. DoD: zero string-interpolated user input in SQL; identifiers whitelisted, values bound. |
| **E-3** | **`SECURITY.md`.** No disclosure policy for an app that stores credentials and is deployed on shared hosting by non-developers. Two paragraphs. |

## 8. Track G — Performance & UX

**Order matters here, and it is counterintuitive.**

| Task | Detail |
|---|---|
| **G-1** | **Pagination — do this *before* #45.** `manage_animals` and `query_animals` contain **zero `LIMIT` clauses**: every animal in the colony is rendered into one HTML table on every load. **This is the actual mechanism behind #45's "pages grow too long."** Server-side pagination + row count + "showing N of M". (Note: with InnoDB, `COUNT(*)` is O(n) — fine at colony scale, but don't count in a loop.) |
| **G-2 (#34)** | **Deterministic cage/animal ordering** in filter selects. One shared ordering helper, not per-page fixes. Watch-out: cage names aren't numeric, so a plain `ORDER BY` gives `C1, C10, C2` — needs a natural-sort key, decided once. |
| **G-3 (#35a)** | **Userbook nav mode.** Most animalbook nav entries are meaningless on userbook pages. `includes/nav.php` is already centralized (148 lines) — contained change. |
| **G-4 (#45)** | **Layout audit → increments.** The issue itself says enumeration is needed. First deliverable is `LAYOUT_AUDIT.md` (per page: failure mode, tasks it must support without scrolling, proposed strategy), *then* per-page work, highest-traffic first. The A/B toggle should be a per-user preference (#35b), not a page switch — persistent trials beat one-click ones. |

## 11. Risks

| Risk | Mitigation |
|---|---|
| **InnoDB conversion verified on MariaDB only** | Confirm on MySQL 8.0.46 before merge. This is exactly the class of surprise the `utf8mb4_0900_ai_ci` incident came from. |
| **Existing production colonies must migrate in place** | Migration script with a mandatory backup-first gate; row counts verified pre/post; downtime scales with colony size. **Open question — see §12.** |
| **`UNIQUE (line, idno)` may fail on existing data** | The duplicates C-2 prevents may already exist. Detect-and-report before constraining; the migration must not fail halfway. |
| **`mb_write()` is a wide refactor** | It touches every write path. Stage it: `manage_animals` first, prove it, then expand. CI grep prevents regression to direct writes. |
| **Deployment order is load-bearing** | Migration SQL before patched PHP goes live. Same rule as M1. |

## 12. Decisions (locked, 2026-07-12)

| Question | Decision | Consequence |
|---|---|---|
| **InnoDB migration?** | **Yes.** Cost accepted (~1.9× disk; 100 MB at 100k animals, 10 MB at real scale). | Track B-1 proceeds; gates C, D, E. |
| **Live installs to migrate?** | **Yes — largest < 5k animals.** | Conversion measured at **< 1 second**. Migration tool stays simple. |
| **Downtime tolerance?** | **Hours (weekend window).** | ~1000× more headroom than needed. No online-DDL complexity. |
| **Who runs the migration?** | **Shell script on the host.** | `migrate_innodb.sh`, matching `setup.sh` conventions. No web UI, no admin-page dependency. |
| **Track A (MVP close-out)?** | **Inside this work block.** | A-1…A-4 land in this scope, not a separate one. |
| **Audit retention/archive?** | **Ships with the feature.** | `audit_log_archive` + growth-rate docs are Track D deliverables, not follow-ups. |
| **DB triggers for audit?** | **Unavailable** (CloudLinux `TRIGGER` grant unreliable). | Audit lives in `mb_write()` (B-3), inside the record transaction (B-1). |
