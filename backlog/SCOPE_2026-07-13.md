## 1a. Findings from the re-baseline (2026-07-13)

**1. B-1 delivered the *engine*, not the *state*.** The merged work shipped a migration framework —
`mb_migrate.sh` (preflight/status/apply/stamp), `migrations/001_innodb_utf8mb4.sh` with a shared charset
guard, `mb_schema_check.sh` + drift baseline, `docs/MIGRATIONS.md`, and a CI job proving convergence and
idempotence on both engines. What it did **not** do was convert the starter SQL or wire the installer to
the ledger, so **a brand-new install still landed on MyISAM/latin1** and stayed there. `docs/MIGRATIONS.md`
already *described* the intended behaviour ("the install schema already reflects every migration"; "it
then calls `stamp`") — the code simply hadn't caught up to the doc. **Closed by B-1b, in this block.**

**2. A charset migration rewrites column *types*, not just charsets.** Converting the starter by hand
(`ENGINE=`, `DEFAULT CHARSET=`) produced a starter that **no migrated database will ever match**:
`CONVERT TO CHARACTER SET utf8mb4` silently widens every blob/text column one tier (`TEXT` →
`MEDIUMTEXT`, `MEDIUMTEXT` → `LONGTEXT`), because `TEXT` is limited by bytes and not characters. Five
columns are affected. **The CI convergence job caught this on its first real use** — which is precisely
the regression class F-1 was built for, arriving one day after F-1 shipped. Rule 7 in
`docs/MIGRATIONS.md` now records it: *derive the starter's state from a real migrated database, never
from what the migration script appears to say.*

**3. B-2 is ~8× larger than scoped.** Not "~18 `new mysqli` sites" — **140 occurrences across 24 files**
(`add_animals.php` alone has 20; `manage_cages.php` 13; `cagecard_printer.php` and `cage_location.php` 10
each). Still **zero `set_charset()` repo-wide**, which is now *more* urgent, not less: the columns are
utf8mb4 as of B-1b, so a connection left at the server default is the last remaining link in the mojibake
chain. Effort revised **M → L**. Staging decision in §12.

**4. `LOCK TABLES` is still live** — 5 sites (`add_animals.php` ×2, `manage_cages.php` ×3), listed under
B-1 in the 07-12 scope and not delivered. It is now a transaction conversion, and it belongs with the
code that replaces the connection: **folded into B-2.**

**5. Track A shrank.** **A-2 (seed data) is already done** — the colony starter ships `INSERT IGNORE`
seeds for a `Limbo` cage location and a `Community` cage role, so a fresh colony *can* accept its first
animal. The install blocker on the 07-12 board no longer exists. A-3's demo endpoints are now *denied* by
the rewritten `.htaccess` (defence-in-depth landed), but the files are still in the tree.

**6. The tracker has no new issues.** Highest number is #55 (a PR). "New issues" for this refresh means
the findings above. Tracker hygiene to do (§13): **#50 (InnoDB) is satisfied → close**; **#46 and #51 are
duplicate audit-trail issues → dedupe**; **#9** remains the only open issue mislabelled `v1.0`.

### B-2 — `mb_connect()`: one connection helper ⬅ **next**

**Decision (§12): one mechanical pass across all 24 files.**

Replace **140** ad-hoc `new mysqli(...)` sites with a single helper that:
- connects,
- **calls `set_charset('utf8mb4')`** — zero occurrences repo-wide today; the last link in the mojibake
  chain now that the columns are utf8mb4,
- sets the per-request actor context for the audit trail,
- centralizes the `debug_mode` block (copy-pasted into ~10 files; renders PHP warnings and SQL to the
  browser when left on in production).

**Folded in from B-1:** convert the 5 `LOCK TABLES` sites to transactions (`add_animals.php:71,178`;
`manage_cages.php:80,412,842`). Note the existing incoherence — `reservations_animals` was *already*
InnoDB yet guarded with `LOCK TABLES`.

**Risk of a single pass, and the mitigation:** the failure mode is a page that silently loses its
connection variable and white-screens. Mitigation is the harness, not caution: HTTP-exercise every page
with a curl cookie jar before delivery, and lint on 8.3. A per-file patch series keeps ordering
flexibility if a page needs to be held back.

### B-3 — `mb_write()`: one write chokepoint

```php
mb_write($conn, $table, $pk_col, $pk_val, array $new_values, $actor, $request_id);
```

Reads the before-image, diffs, writes the record **and** its audit rows in **one transaction**. Unchanged
columns are neither written nor logged. Precondition for #27a (one place to check the version token), the
audit trail (one place to log), and CSRF (one place to verify the token). **Build it once, deliberately.**

**Exit criteria for Track B:** every colony table is InnoDB on both engines *(met for fresh installs;
per-install after `apply`)*; every page connects through `mb_connect()`; a non-ASCII strain name
round-trips correctly; `mb_write()` exists and `manage_animals` uses it.

## 6. Track D — Audit trail (#46 / #51 — dedupe first)

Full design in `ISSUE_audit_trail.md`. DB triggers are **unavailable** (the `TRIGGER` privilege can't be
relied on under CloudLinux shared hosting), so the audit write lives in `mb_write()` (B-3), inside the
same transaction as the record change (now genuinely available, per B-1/B-1b).

Interned actor/field lookups + `binary(16)` request id + two indexes = **193 MB per 1M change rows** (vs
305 MB naive). **The audit log will outgrow the colony it describes.** Ship the archive step and the
growth-rate documentation *with* the feature, not after.

Sequence: tables + INSERT-only grant → `mb_write()` logging → `manage_animals` first (same pass as C-1) →
remaining tables → per-record History panel → admin Audit page → export → archive.
